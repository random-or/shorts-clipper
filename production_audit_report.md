# Shorts Clipper Production Audit Report

This document contains the findings of the exhaustive 10-phase production audit performed on the `shorts-clipper` repository.

## PHASE 1: REPOSITORY INVENTORY

The repository implements a sophisticated, multi-process architecture tailored for automated video processing.

*   **Directory Structure**: Modularized into distinct concerns: `core`, `pipeline`, `scout`, `transcription`, `downloader`, `analyze`, `highlight_detection`, `publish`, `render` (thumbnailer), `rendering` (ffmpeg/crop), `captions`, `cropping`, `social`, `providers`, `api`, `ui`.
*   **Core Modules**: 
    *   `core.models`: Defines immutable dataclasses (`TranscriptWord`, `TranscriptSegment`, `ClipWindow`, `HighlightScore`, `Job`).
    *   `core.queue`: SQLite-backed persistent job queue tracking execution states.
    *   `core.worker`: Decoupled background task worker for offloading heavy jobs.
*   **Pipeline Stages**: Orchestrated by `pipeline/runner.py`.
*   **Entry Points**: 
    *   `__main__.py`: CLI interface.
    *   `api/server.py`: FastAPI server for the Vanguard web dashboard.
    *   `core/worker.py`: Background worker process.
*   **Background Workers**: Polling worker system that grabs jobs from `jobs.db`. Also features an Autopilot Watchdog thread that polls channels for new uploads.
*   **Database Models**: SQLite-backed local databases using WAL mode: `jobs.db`, `scout_memory.db` (for ML-like historical success tracking), and `feedback.db`.
*   **Configs & Env**: `core/settings.py` parses 19 configuration variables. 
*   **Tests**: 5 test files (`test_foundations.py`, `test_new_modules.py`, `test_scout_v2.py`, `test_rendering.py`, `test_api_fixes.py`) using a mix of `unittest` and `pytest`.

## PHASE 2: PIPELINE TRACE

The core execution flow (`run_autopilot` → `run(url)`) follows a highly optimized, single-pass pipeline:

1.  **SCOUT**: `get_trending_link()` queries YouTube (via Data API or `yt-dlp`), filters candidates by age/duration/views, scores them, and deeply evaluates the top 5 finalists.
2.  **TRANSCRIBE**: System checks cache → attempts native YouTube subtitle extraction → falls back to `transcribe_clip()` (using Gemini 2.5 Flash for speed, or local `faster-whisper`).
3.  **SELECT**: `GeminiProvider.select_multiple_clips_detailed()` processes the transcript and selects the most viral segment(s).
4.  **CLIP & CROP**: `process_to_vertical()` and `stream_render_pipeline()` in `rendering/pipe.py`. Notably, this pipes `yt-dlp` output directly into `FFmpeg` stdin, performing crop, scale, and subtitle burning *in memory* without intermediate disk writes.
5.  **SUBTITLE**: `burn_subtitles()` generates `.ass` style files and injects them via FFmpeg. Includes features like word-level highlighting and micro-animations for emotional trigger words.
6.  **PUBLISH**: Metadata (Title, Description, Tags) is generated by Gemini and saved to a sidecar JSON file.

## PHASE 3: QUALITY SYSTEM AUDIT

The pipeline utilizes a sophisticated, 3-layer hybrid scoring system:

1.  **Intermediate Scout Score**: A capped, multi-dimensional score evaluating *Velocity* (max 15), *Engagement* (max 15), *Recency* (max 10), *Momentum* (max 5), and *Channel History* (max 15).
2.  **Rule-Based Highlight Score**: Scrutinizes transcripts for hook patterns, emotional trigger words, pacing, and caption density.
3.  **Gemini AI Evaluation**: Grades clips on a 0-100 scale based on emotional peaks, clip-ability, and niche relevance. **Hard Threshold: >= 85**. Clips scoring lower are rejected, triggering fallback mechanics.
4.  **Feedback Loop**: `analyze/feedback.py` stores historical clip performance to calculate a weighted `performance_score` that influences future scout decisions.

## PHASE 4: POSTABILITY READINESS

*   **Output Quality**: Yes, the system generates ready-to-post, fully rendered vertical `.mp4` video files.
*   **Assets**: Alongside the video, it generates a `.jpg` thumbnail (at the 25% timestamp hook point) and a sidecar `.json` containing AI-generated viral titles, hashtags, and descriptions.

## PHASE 5: MULTI-CANDIDATE ANALYSIS

*   **Comparison Engine**: Scout V2 does not just grab the first video. It builds an initial pool of discovered URLs, trims it to the top 15 via the Intermediate Score, and runs a deep execution pass (transcription + AI evaluation) on the top 5 candidates. The system then selects the single candidate with the highest composite score.

## PHASE 6: YOUTUBE PUBLISHING AUDIT

*   **Publishing Adapters**: The files in `publish/` (`youtube.py`, `instagram.py`) are currently just **stubs** that log success.
*   **Actual Upload Logic**: The real upload engine lives in `social/youtube.py` (`upload_short()`). It uses the official `google-api-python-client` to perform chunked, resumable uploads to YouTube Shorts.
*   **OAuth Flow**: A complete, robust OAuth 2.0 flow is implemented in `api/server.py` (`/api/youtube/connect` & `/api/youtube/callback`). It correctly handles dynamic redirect URIs for local vs. cloud (HuggingFace) deployments and persists the token in `.cache/shorts-clipper/token.pickle`.

## PHASE 7: TEST COVERAGE AUDIT

*   **Execution Status**: The test suite was successfully executed using the virtual environment. All 46 tests passed.
*   **Blind Spots**: There are **4 explicit stub tests** in `tests/test_new_modules.py`:
    *   `test_is_suitable_enforces_max_age_days`
    *   `test_scout_by_niche_rotation`
    *   `test_scout_by_keyword_multi_platform`
    *   `test_is_suitable_rejects_low_resolution`
    These currently just contain `pass` and do not test the actual filtering logic.

## PHASE 8: PRODUCTION READINESS

*   **Strengths**: Extremely robust process management. The worker system gracefully handles job cancellation by monkey-patching `subprocess.Popen` to track and kill orphaned child processes. Network failures (429/403) are mitigated by exponential backoff and circuit breakers. The stream-rendering pipeline minimizes disk I/O, allowing it to run on low-resource cloud instances.
*   **Weaknesses**: 
    *   **Security Risk**: The local `.env` file contains hardcoded, live `GEMINI_API_KEY` and `YOUTUBE_API_KEY` credentials.
    *   **Architectural Debt**: `api/server.py` is a monolithic 1,200-line file that handles routing, OAuth, job dispatching, and UI serving.
    *   **UI**: The frontend is a massive 198KB inline HTML file, which will be difficult to maintain.

## PHASE 9: NEXT BEST INVESTMENT

Based on the audit, the following engineering tasks should be prioritized:

1.  **Security Hotfix**: Immediately scrub the live API keys from the `.env` file and rotate them.
2.  **Test Implementation**: Replace the 4 `pass` stubs in `test_new_modules.py` with real assertions to verify the Scout filtering constraints.
3.  **Refactoring**: Break apart `api/server.py` into FastAPI sub-routers (e.g., `routers/jobs.py`, `routers/youtube.py`, `routers/scout.py`).
4.  **Adapter Cleanup**: Wire the stubs in the `publish/` directory to interface with the actual implementations in `social/youtube.py`.

## PHASE 10: FINAL VERDICT

**Status: READY FOR PRODUCTION WITH MINOR REFACTORING**

The `shorts-clipper` repository is a highly sophisticated, production-grade video automation pipeline. It elegantly solves complex problems (like asynchronous job handling, dynamic crop tracking, and rate-limiting) while optimizing for compute efficiency (via piped FFmpeg streams and Gemini Flash fallbacks). Once the minor testing gaps are filled and the monolithic API server is factored out, this system will be exceptionally stable.
